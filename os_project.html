Bootstrapping & Cross-compilation

1.1 Definitions

Our goal here is to build a GCC cross-compiler for the i386-elf target. Therefore, I have to build binutils, gdb and gcc myself to set them for this configuration.

Bootstrapping is a process in which we use an older version of a compiler to compile its new version. Hence, we start with a basic compiler (compiled with a native compiler) up to buidling a more advanced (and maybe self-hosted) compiler. Bootstrapping is very interesting as it leverages previous features and achievements of a compiler to create a new and more advanced version of it. Here is an illustration with the T-diagram: A self-hosted compiler is a compiler designed to compile its source programmation language. A cross-compiler is a compiler whose purpose is to build executable programs (binary files) which are aimed at another OS or architecture than the one we are working on. For example, if we work on a Linux environment but we want to create executables for Windows whose OS (and maybe architecture) differs, we can use a cross-compiler (such as minGW-64) to do so.

As an example, the GCC compiler is a cross-compiler that relies on bootstrapping. Thus, it can be configured to generate code on various targets and architectures (using minGW-64 for Windows OS on a x64 architecture). On top of that, it supports a wide range of source langages such as C, C++, Fortran, Go, Pascal, Rust, etc. This is due to its structure, which allows it to separate front-end (source syntax) and back-end (target syntax) code.

First, start by setting some environment variables to make our lives easier. TARGET is the target architecture, PREFIX is the path to your compiler's root directory. This path can be changed. Finally, PATH is the path to the compiler binaries (or executable).

export TARGET=i686-elf
export PREFIX="$HOME/opt/cross"
export PATH="PREFIX/bin:$PATH"
Compilation of Binutils

Download these Binutils packages in the $HOME/src directory.

Compilation of GDB
GDB packages

Compilation of GCC

Clone the GCC repository. For more detailed information, read through this page.

HOME/src$ git clone https://gcc.gnu.org/git/gcc.git
Once downloaded, make sure to download prerequisites (GMP, MPFR, ISL and MPC libraries).
tar xzf gcc-x.y.z.tar.gz
cd gcc-x.y.z
./contrib/download_prerequisites
cd ..
mkdir build-gcc
cd build-gcc
$../gcc-x.y.z/configure --target=$TARGET --prefix=$PREFIX --enable-languages=c,c++
make make install
HOME/src/gdb$ ./contrib/download_prerequisites
Finally, configure and build our GCC cross-compiler for the i686-elf target.
HOME/src/build-gdb$ ../gcc/configure --target=$TARGET --prefix="$PREFIX" --disable-nls --enable-languages=c,c++ --without-headers

Kernel loading and booting

A computer mainly uses 3 components to boot: the BIOS, the bootloader and the kernel. When we turn on a computer, the BIOS (Basic Input/Output System) is the first program to be loaded into the firmware RAM. The CP register points to the BIOS code and executes it. The BIOS then loads the bootloader into the RAM by fetching the first sector of the boot device (usually the hard drive) and jumping to it.

The bootloader is a small program responsible for loading the kernel into the RAM. It contains various information about the kernel such as its location on the disk, its size, etc. In this project, I will be using GRUB as bootloader.

Finally, the bootloader loads the kernel into the RAM and jumps to it. After this, one problem remains: the stack pointer (ESP) is not initialized. However, C++ programs expect this pointer to be initialized when the program starts. To initialize it, we need to create a assembly file (loader.s) that will initialize this pointer and jump to the kernel's entry point. On the other hand, the kernel program will be written in C++ language. Hence, we will end up with 2 files: a loader.s file and a kernel.cpp file. Since those are from different natures (assembler and C++), we need to find a way to "merge" them. To do that, we will be using a linker. As a result, we will build both files into object files (.o) and link them using the linker program ld. All of this is discussed further. ELF format

There exist different types of binary formats. .bin files are raw binary files. .elf (Executable and Linkable Format) files are more sophisticated as they contain machine code and also extra information about the program (such as the entry point, the size of the code, etc.) that aids debugging and analysis.

Loader file

There are some issues we need to sort out and that require assemby code. Firstly, the stack pointer has to be initialized. Secondly, when the bootloader loads the kernel into the RAM it searches for a so-called Magic number that informs it that the kernel.bin file is a valid kernel.

A Magic number is a sequence of few bytes likely located at the beginning of a file which are unique to a particular file type. They are are used as a file signature by system. It helps the system to differentiate between different file types.

Thirdly, the bootloader will also make the EAX register point to the Multiboot information structure and the EBX register point to the Magic number. Knowing that, we could benefit from this to store those values by pushing them on the kernel's stack. Eventually, the loader.s file's goals are:

  • to initialize the stack pointer for the kernel
  • to write the Magic number: 0x1BADB002
  • to push the EAX and EBX registers on the stack
